강의가 VirtualBox 기준이라, 이미 쓰던 WSL2와는 별도로 VirtualBox의 Ubuntu 22.04에 ROS2 Humble 환경을 새로 만들었어요. 로케일부터 apt 저장소, desktop 메타패키지, `ros2 doctor`까지 설치 문서를 따라 검증하고, 강의 워크스페이스를 `colcon build`로 빌드했습니다. 그 과정에서 빌드가 두 번 멈췄는데, 둘 다 원인이 소스 코드가 아니라 **환경과 상태**였다는 게 오늘의 요점이에요.

## 시스템에 없는 라이브러리

첫 실패는 `ros_serial_c` 패키지에서 났어요.

```
--- stderr: ros_serial_c
CMake Error at CMakeLists.txt:14 (pkg_check_modules):
  A required package was not found
```

14번째 줄은 `pkg_check_modules(LIBSERIAL REQUIRED libserial)`였습니다. 시리얼 통신 라이브러리 `libserial`을 시스템에서 찾다가 못 찾은 거예요. 소스는 `git clone`으로 받아 다 있는데, 정작 그 소스가 링크하려는 **시스템 라이브러리가 안 깔려 있던** 상황이었습니다. `sudo apt install -y libserial-dev`로 설치하고 다시 빌드하니 통과했어요. 빌드 중에 뜬 `unused variable 'timeout_ms'`는 경고라 무시해도 되는 줄이었고요.

여기서 배운 건 `pkg_check_modules`의 성격이에요. 이건 CMake가 `pkg-config`를 통해 **시스템에 이미 설치된 라이브러리**를 찾는 방식이라, 소스 트리 안에 없는 외부 의존성입니다. 그래서 남의 리포를 clone해 빌드할 때 이런 시스템 패키지 요구가 자주 걸려요. ROS에는 이걸 자동으로 훑어 깔아 주는 `rosdep`이 있는데(`rosdep install --from-paths src`), 빌드 전에 한 번 돌려 두면 이런 종류의 실패를 미리 막습니다.

## 옛 경로를 기억하는 빌드 캐시

두 번째 실패는 더 헷갈렸어요. 분명 잘 있는 패키지인데 소스 디렉토리가 없다고 나왔거든요.

```
CMake Error: The source directory ".../ROS2_basic_study/cv_msg" does not exist
```

원인은 제가 소스를 `ROS2_basic_study/` 폴더에서 워크스페이스의 `src/`로 옮긴 데 있었습니다. 옮기기 전에 한 번 빌드한 적이 있어서, `build/` 폴더에 남은 `CMakeCache.txt`가 **옛 절대경로**를 그대로 기억하고 있었어요. colcon은 처음 빌드할 때 각 패키지의 소스 위치를 절대경로로 캐시하는데, 소스를 옮기면 그 캐시가 낡아서(stale) 사라진 경로를 계속 가리킵니다. `build/`, `install/`, `log/` 세 폴더를 지우고 클린 빌드하니 26개 패키지가 전부 성공했어요.

이 세 폴더는 어차피 colcon이 만들어 내는 산출물이라 버전 관리에 넣지 않는 것들이에요. 그래서 소스를 옮기거나 이름을 바꾼 뒤에는 망설임 없이 지우고 다시 빌드하는 게 맞습니다. "방금까지 되던 빌드가 갑자기 없는 경로를 찾는다"면 캐시가 낡았을 확률이 높아요.

두 실패의 공통점은 분명했습니다. 빌드가 깨진 자리는 소스 코드가 아니라, 무엇이 설치돼 있는가(환경)와 캐시가 최신인가(상태)였어요. `pkg_check_modules`의 에러 메시지도 `CMakeCache.txt`가 붙든 옛 경로도, 코드를 아무리 읽어도 안 보이는 정보입니다. 그래서 빌드 로그가 코드 줄을 가리켜도, 코드를 고치기 전에 환경과 상태부터 확인하는 순서를 몸에 붙이게 됐어요.
