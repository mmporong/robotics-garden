로봇에는 모니터도 키보드도 없어서 내 PC에서 원격으로 붙어 작업해요. 그런데 이 환경에서 생기는 문제는 대부분 코드에 에러가 안 남아요. 프로그램이 그냥 없어지거나, 잘 도는 것처럼 보이는데 데이터만 안 들어와요.

강의에서 들은 현장 문제 두 가지와, 예습 자료를 준비하며 문서로 확인한 한 가지를 정리했어요. 아직 직접 겪어본 것은 아니고, 환경을 갖추기 전에 미리 알아두려고 모은 것이에요.

## 원격 접속은 두 가지를 씁니다

SSH(Secure Shell)는 터미널만 넘겨줘요. 글자만 주고받으니 가볍고 끊길 위험이 적어서, 명령 실행과 노드 구동은 대부분 여기서 해요. VNC(Virtual Network Computing)는 화면 전체를 전송해요. rviz2나 Gazebo처럼 그림을 봐야 하는 작업에만 쓰고, 무거워서 상시로 열어두진 않아요.

로봇 안에서 도는 컴퓨터는 SBC(Single Board Computer)예요. 라즈베리파이나 젯슨 같은 작은 컴퓨터에 Linux가 올라가 있고, ROS는 그 위에서 돌아요. 여기서 첫 번째 함정이 나와요. 이 컴퓨터는 내 개발 PC보다 훨씬 약하고, 무선으로 연결돼 있어요.

## 접속이 끊기면 프로그램도 같이 죽어요

로봇이 움직이면 와이파이 신호가 약해지는 구간이 생겨요. 이때 SSH 세션이 끊기는데, 문제는 그 세션에서 띄운 프로그램도 함께 종료된다는 점이에요. 셸이 종료되면 자식 프로세스에 종료 신호가 전달되기 때문이에요.

주행 중이라면 제어 노드가 죽은 채로 로봇이 굴러가요. 로그를 봐도 프로그램이 정상 종료된 것처럼 보이고, 다시 접속하면 아무것도 안 돌고 있어요. 코드를 아무리 봐도 원인이 없어요.

대응은 세션과 프로세스를 분리하는 거예요. `tmux`나 `screen`으로 프로그램을 띄우면 접속이 끊겨도 로봇 안에서 계속 돌아요. 다시 붙어서 `tmux attach`로 원래 화면으로 돌아갈 수 있어요.

```bash
tmux new -s robot          # 세션 만들고 그 안에서 노드 실행
tmux attach -t robot       # 재접속 후 원래 화면으로
```

## 메모리가 부족하면 OS가 골라서 죽여요

SBC는 메모리가 작아요. 노드를 여럿 띄우고 rosbag 기록까지 걸면 금방 차요. Linux는 메모리가 바닥나면 OOM Killer(Out Of Memory Killer)를 동작시켜 프로세스를 강제 종료하는데, 선택 기준이 메모리를 많이 쓰는 쪽이에요. 그래서 하필 제일 중요한 노드가 먼저 죽는 경우가 많아요.

증상이 SSH 종료와 비슷해요. 에러 로그 없이 노드가 사라져요. 다만 원인이 다르니 확인하는 곳도 달라요. 커널이 남긴 기록을 보면 돼요.

```bash
dmesg | grep -i "killed process"
# Out of memory: Killed process 1234 (my_node) total-vm:...
```

여기에 이름이 찍혀 있으면 내 코드 문제가 아니라 자원 문제예요. 노드를 나눠 띄우거나, rosbag 기록 항목을 줄이거나, 이미지 같은 큰 메시지의 큐 크기를 줄이는 쪽으로 대응해요.

## 토픽은 도는데 콜백만 안 불려요

세 번째는 ROS2 고유의 문제예요. LiDAR 구독 노드를 예습용으로 준비하면서 자료에서 확인한 지점이에요.

ROS2에는 QoS(Quality of Service)라는 통신 품질 설정이 있어요. 그중 신뢰성(reliability) 항목이 두 값을 가져요. RELIABLE은 도착을 보장하려고 재전송하고, BEST_EFFORT는 유실을 허용하는 대신 지연을 줄여요. 센서 드라이버는 대부분 BEST_EFFORT로 발행해요. 초당 수십 장씩 오는 스캔 데이터는 한 장 놓쳐도 다음 장이 오니까요.

문제는 구독 쪽 기본값이 RELIABLE이라는 점이에요. 발행자와 구독자의 QoS가 호환되지 않으면 연결 자체가 성립하지 않고, ROS2는 이걸 에러로 알려주지 않아요. 결과가 이렇게 나와요.

```bash
ros2 topic list          # /scan 이 보여요
ros2 topic hz /scan      # 초당 10회로 잘 돌아요
# 그런데 내 노드의 콜백만 한 번도 안 불려요
```

진단은 발행자와 구독자의 QoS를 직접 비교하는 거예요.

```bash
ros2 topic info /scan --verbose
# Publisher / Subscription 각각의 Reliability 값을 확인해요
```

해결은 센서용 프리셋을 쓰는 거예요. `rclcpp::SensorDataQoS()`가 BEST_EFFORT에 KEEP_LAST(5)를 묶어둔 설정이라, 센서 토픽을 구독할 때는 이걸 쓰면 대부분 맞아요.

```cpp
sub_ = this->create_subscription<sensor_msgs::msg::LaserScan>(
    "scan", rclcpp::SensorDataQoS(),
    [this](const sensor_msgs::msg::LaserScan& msg) { /* ... */ });
```

## 세 가지의 공통점

셋 다 내 코드에 에러가 없어요. 컴파일도 되고 실행도 되는데 결과만 안 나와요. 그래서 코드를 들여다보는 동안에는 원인이 안 나오고, 봐야 할 곳이 각각 따로 있어요. 세션이 살아 있는지, 커널이 프로세스를 죽였는지, 발행자와 구독자의 설정이 맞는지예요.

로봇 소프트웨어에서 디버깅이 어려운 이유가 여기에 있어요. 움직이는 중에는 브레이크포인트를 걸 수 없고, 멈추면 그 상태가 사라져요. 그래서 실무에서는 `ros2 bag`으로 주행을 통째로 기록해두고 나중에 재생하면서 분석한다고 해요. 같은 상황을 몇 번이고 다시 볼 수 있다는 게 무기가 되니까요.

증상만 보면 셋이 거의 구분이 안 되니, 확인하는 명령을 미리 손에 익혀두는 쪽이 나아 보여요. 환경을 갖추고 나서 실제로 어느 쪽이었는지 확인해볼 생각이에요.
